## Relatório admin diário não chegou (23/06) — causa + retry na fila
- **Sintoma (reportado pelo Arthur):** relatório admin não chegou no grupo da Barra ontem. Investiguei: a fila `fila_relatorios_whatsapp` mostrava **os 3** (Barra 20:00, CG 20:01, Recreio 20:02) com `status='erro'` e `erro="Session status is not as expected..."` — a sessão WAHA da caixa **Sol** estava fora/instável às 20h. O cron `relatorio-diario-20h` (job 13) **disparou OK** (status `succeeded`); a falha foi na **entrega**.
- **Bug de design:** a fila **não tinha retry** — falhou 1x → `erro` e parou (a query só pegava `pendente`). Sessão voltou, ninguém reenviou.
- **Fix — retry seguro em `processar-mensagens-agendadas` v12** (deploy MCP, `verify_jwt=false`), função `processarFilaRelatorios`:
  - Migration `fila_relatorios_retry_colunas`: `tentativas int default 0` + `ultima_tentativa_em timestamptz`.
  - **Nunca envia 2x:** `enviada` (sucesso) e `falhou` (erro terminal) nunca são repescados. Só reenvia (`erro`) quando a falha indica que a msg **não saiu** (`relatorioErroRetryavel`: session/connection/timeout/50x). Erro do WhatsApp **após** receber a requisição → `falhou` (não reenvia, pra não duplicar).
  - **Guarda extra:** antes de enviar (WAHA), confere `GET /api/sessions/{session}` == `WORKING`; sessão fora → nem tenta, fica retryável.
  - Limite `RELATORIO_MAX_TENTATIVAS=8`; janela de retry **2h** (`agendada_para >= now()-2h`) → não reenvia relatório de dias anteriores; destrava itens presos em `enviando` há +5min.
  - Validado: query de candidatos retorna vazio agora (os 3 de ontem ficam fora da janela → **não reenviados**).
- **Erro 463 (Daiana, notificação 360) é problema SEPARADO:** trava anti-spam do WhatsApp ("reach-out time-lock" / `tctoken`) p/ contato 1:1 **sem conversa prévia** + bug do engine **GOWS** do WAHA (caixa Sol, `2026.5.1`). Sessão sã (envio pra grupo/contato com histórico funciona). Saída rápida: contato manda 1 msg pra Sol antes; fundo: trocar engine GOWS→NOWEB/WEBJS ou atualizar WAHA.
